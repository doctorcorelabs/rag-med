# Workflow Medical RAG (v3.1.0)

Dokumen ini menjelaskan alur kerja teknis sistem Medical RAG, mencakup integrasi antara lingkungan lokal (Python) dan lingkungan produki (Cloudflare Workers).

## 1. Arsitektur Umum
Sistem ini menggunakan pendekatan **Hybrid RAG** dengan dua jalur pemrosesan data utama:

- **Local Development**: FastAPI (Python) + SQLite + Local Indexing.
- **Production (Deploy)**: Cloudflare Workers (Hono.js) + Supabase (PostgreSQL/Vector) + R2 Storage.

---

## 2. Alur Pencarian Materi Klinis (Deep Search)
*Contoh: "Jelaskan tatalaksana Gagal Jantung"*

1.  **Intent Detection**: Sistem mengenali entitas penyakit (`disease_name`) dan topik spesifik (`intent_category`).
2.  **Hybrid Retrieval**:
    - **Vector Search**: Mengambil konteks berdasarkan kesamaan semantik.
    - **BM25 / FTS**: Mengambil konteks berdasarkan kecocokan kata kunci (keyword matching).
3.  **RRF Fusion**: Menggabungkan hasil pencarian untuk mendapatkan 8-15 referensi (`evidence`) terbaik.
4.  **AI Synthesis (GitHub Copilot)**:
    - AI melakukan *self-reflection* (verifikasi fakta dari dokumen).
    - AI menyusun jawaban terstruktur dengan format Markdown.
    - **Citations**: Setiap klaim disertai sitasi granul `[E1], [E2]` yang merujuk pada file/halaman asli.

---

## 3. Alur Daftar Penyakit (Direct DB Listing)
*Contoh: "Daftar penyakit kardiovaskular dari Atria"*

1.  **Bypass Vector Search**: Jika terdeteksi permintaan daftar/katalog, sistem melewati jalur pemrosesan vektor yang lambat dan terbatas.
2.  **Direct SQL Query**: Melakukan query `SELECT DISTINCT` langsung ke tabel `chunks` untuk mengambil seluruh judul topik.
3.  **Source Filtering**: Memfilter secara spesifik berdasarkan kata kunci sumber (misal: Atria, Mediko, Kaplan).
4.  **AI Pure List Filtering**: Menggunakan model AI (`askCopilotForPureList`) untuk membersihkan daftar mentah dari database:
    - Menghapus angka halaman, simbol, atau judul bab non-medis.
    - Menyisakan hanya daftar murni nama penyakit/kondisi medis.
5.  **Output**: Memberikan daftar lengkap yang komprehensif tanpa risiko terpotong (truncation).

## 4. Struktur Database (Supabase / SQLite)
| Kolom | Deskripsi |
| :--- | :--- |
| `source_name` | Nama file/sumber asli (misal: A1 Atria Respi). |
| `page_no` | Nomor halaman dokumen asli. |
| `heading` | Judul topik/penyakit. |
| `content` | Isi teks mentah (chunking). |
| `embedding` | Representasi vektor (hanya di Supabase/Chroma). |
| `stase_slug` | Kategori stase (misal: IPD, Anak). |

## 5. Sinkronisasi Deploy
Setiap perubahan logika di file Python (`retriever.py`, `copilot_client.py`) harus di-porting ke file TypeScript di folder `worker/src/`:
- `api.py` -> `index.ts`
- `retriever.py` -> `retriever.ts`
- `copilot_client.py` -> `copilot-client.ts`

Untuk melakukan deploy ulang setelah perubahan:
```bash
cd worker
npm run deploy
```
